Teaching JavaScript to newer developers made one thing obvious very quickly: code that is hard to explain is often harder than it needs to be.

When students struggle, it is not always because the topic is advanced. Sometimes the real problem is that the code path contains too many hidden assumptions.

Teaching pushed me to pay more attention to:

- naming that reduces guesswork
- functions with one obvious job
- data flow that can be described in a sentence
- examples that expose edge cases early

This carries back into product work. Clear code is easier to review, easier to change, and easier to trust under pressure.

I still like elegant solutions, but I trust explainable ones more. If a teammate or student can follow the reasoning without ceremony, the implementation usually has a stronger foundation.
